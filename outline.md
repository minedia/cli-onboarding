## 0. 講座メッセージ（最重要）

### 一行で言うと

> **AIエージェント時代に、CLIを知る人と知らない人の生産性は16倍以上違う。**
> その差は毎月毎月、複利で広がっていく。

### なぜ「今」なのか

過去：CLI は "エンジニアの道具"。一般職には不要だった。
**今：CLI は "AIに仕事を任せる全員の道具"。**

AIエージェント（Claude Code、Cursor、各種MCP）は、GUIではなくテキスト＝コマンドラインを介して動く。AIに業務を委任しようとすると、必ずどこかで「ターミナル」にぶつかる。

- AIに「このCSVを整形して」と頼む → AIはスクリプトを書く → **動かすのはあなた（CLI）**
- AIに「画像を一括変換して」と頼む → AIはコマンドを返す → **コピペして実行するのはあなた（CLI）**
- Claude Codeを契約した → **起動・操作・ファイル指定が全部CLI**

「コマンドプロンプトに貼って実行してください」で止まる人と、自分で改造して定期実行まで持っていく人の差。これがAI時代の新しい "PCリテラシー" になっている。

### 格差の具体イメージ（オープニングでスライドに出す数字）

| | CLIを使えない人 | CLIを使える人 |
|---|---|---|
| Claude Codeで自動化できる業務 | 月3時間 | 月50時間 |
| 100ファイルのリネーム | 2時間（手作業） | 3秒（1コマンド） |
| AIスクリプトをもらった時 | 「動かし方がわからない」 | 即実行・改造・定期化 |
| 同じツール、同じ料金、同じ給料 | 生産性 ×1 | 生産性 **×16** |

> ※「月50時間」「16倍」は感覚値。受講者が"自分ごと"として体感できる数字感を狙う。実数は講演時に「私（CTO）の体感では…」と前置きする。

### さらに：Claude Code の "裏側" もまったく違う

AIエージェントは、あなたの環境次第で**裏側でやっていることが根本的に変わる**。
同じ「このプロジェクトから TODO を全部探して」という依頼でも：

| | CLI環境（あなたがCLIを使える） | GUI環境（CLIが分からない） |
|---|---|---|
| AI内部の動作 | `grep -r "TODO" .` を実行して結果を読む | デスクトップのスクショ撮影 → 画像解析 → 座標特定 → クリック → またスクショ → ... の延々と繰り返し |
| 所要時間 | **0.5 秒** | 数分〜十数分 |
| トークン消費 | 約 1,000 トークン | **数十万〜数百万トークン**（数百倍） |
| 失敗時の回復 | コマンドを直して即再実行 | 何が起きたか不明・最初からやり直し |
| あなたが理解できるか | コマンドが読めるから追える | ブラックボックス |

つまり、**CLIが使える環境を持っていることそれ自体が、AIの "燃費" と "賢さ" を桁違いに上げる**。
あなたのPCがCLI環境なら、AIは省エネで賢く動く。GUIしかなければ、同じAIが数百倍のコストで、それでも手探りで動く。

> これがAI時代に「CLIを知っている人ほど、AIに任せられる量が指数関数的に増える」という構造的な理由。

### メタファ（オープニングで使う）

> **AIエージェントは「英語しか喋れない、超優秀なアシスタント」。CLIはその英語そのもの。**
> 英語が読み書きできない人は、せっかくの優秀な助手に1ミリも仕事を頼めない。
> 今日の60分は、その「英語」のいちばん基礎を体に入れる時間。

### 2. シェルとコマンドの基礎（9分）★ハンズオン②

**話すポイント**

**(1) シェルとは（30秒）**
- シェル = 「あなた」と「PC」の間に立つ通訳
- あなたが日本語/コマンドで指示 → シェルが翻訳 → PCが動く

**(2) コマンドのフォーマット — これが全ての基本（2分）**

> **CLIで打つもの = "スペースで区切った単語の列"。それ以上でも以下でもない。**

ルールはたった2つ：

1. **一番左の単語 = コマンド名**（何をするか）
2. **それより右の単語 = 全部そのコマンドへの "追加指示"（オプション / 引数）**

```
ls   -la   /Users
└┬┘  └─┬─┘  └─┬──┘
 │     │     └── 「どこの」を指定する追加指示
 │     └──────── 「どう表示するか」を指定する追加指示（- や -- で始まる）
 └────────────── コマンド名（一覧表示せよ）
```

ポイント：
- スペースで区切る。タブやカンマではない
- 右側は何個並べてもOK。順序もコマンドによる
- 「右側に何が書けるか」はコマンドごとに違う

**(3) `--help` の慣習（1分）**

> **「このコマンド、何ができるんだっけ？」と思ったら、まず `--help` を付ける。**

```
ls --help
git --help
python --help
```

ほぼ全てのコマンドが、慣習的に `--help` で使い方を表示してくれる。
**辞書を引くようにコマンドを学べる**。これを知っているだけで、未知のコマンドが怖くなくなる。

- Mac / Git Bash では `man ls` も使える（詳しいマニュアル）
- 終了は `q` キー

**ハンズオン②：基本コマンド5つを叩く + `--help` を試す**

| コマンド | 意味 | 試す例 |
|---|---|---|
| `pwd` | 今いる場所を表示 | `pwd` |
| `ls` | 中身を一覧 | `ls`、`ls -la` |
| `cd` | 場所を移動 | `cd ~`、`cd Desktop` |
| `echo` | 文字を表示 | `echo Hello, World` |
| `cat` | ファイル内容を表示 | （後でファイルを作ってから） |

最後に必ず：

```bash
ls --help
```

→ 大量のオプションが出てくる。**全部覚えなくていい**。「ここを見ればいい」と知ることが大事。

**Windows補足**
- Git Bash 前提なので、Mac と**まったく同じコマンド**が動く
- 補足説明は不要（これが Git Bash 統一のメリット）

**避けるべき脱線**
- PowerShell の独自記法 → 触れない
- パイプ `|` とリダイレクト `>` → 応用編で紹介、ハンズオンでは触れない
- 「引数」と「オプション」の用語の厳密な使い分け → 初心者には「右側は全部追加指示」で十分

**FAQ想定**
- Q「タブで補完できる？」→ A「できます！半角入力中にTabキー、めちゃくちゃ便利。今日中に必ず使ってください」
- Q「上下キーで履歴が出る？」→ A「出ます。↑キーで一個前のコマンドを呼び出せる」
- Q「`-la` と `-l -a` は同じ？」→ A「同じ。短いオプションは繋げて書ける慣習」

**(4) ショートカットキー — 命綱（1分）★**

ハンズオン②の後、必ず触れる：

| キー | 効果 | 重要度 |
|---|---|---|
| `Ctrl + C` | 実行中のコマンドを止める | ★★★ |
| `Ctrl + L` | 画面をクリア | ★★ |
| `Ctrl + R` | 履歴をインクリメンタル検索 | ★★ |
| `Ctrl + A` / `Ctrl + E` | 行頭 / 行末へ | ★ |
| `Ctrl + D` | ターミナルを閉じる | ★ |

実演手順：
1. `sleep 100` を打って実行
2. 動かないように見える状況を見せる
3. `Ctrl + C` で止める → 「これ知らないと、ハマったときにパニックになります」
4. `Ctrl + R` を1回だけ実演（`ls` などで履歴検索デモ）

### 3. ファイル・ディレクトリ操作（13分）★ハンズオン③ — 最重要

**話すポイント**
- **カレントディレクトリ** = 「今いる場所」
  - メタファ：あなたの「立ち位置」。すべてのコマンドはここを基準に動く
- **ホームディレクトリ** = 「自分の家」
  - Windowsなら `C:\Users\<あなたの名前>`
  - Macなら `/Users/<あなたの名前>`
- **チルダ `~` の説明（必ず触れる）**
  - 読み方：**「チルダ」**（英語: tilde）
  - 意味：ホームディレクトリの省略記号
  - `cd ~` でどこからでも一発でホームに戻れる
  - `cd ~/Documents` のようにホーム起点の道順も書ける
  - ターミナルのプロンプトに `~` と表示されていたら「今ホームにいる」のサイン
- **絶対パスと相対パス**
  - 絶対パス：`/Users/yuki/Desktop/memo.txt`（住所をフルで書く）
  - 相対パス：`Desktop/memo.txt`（現在地からの道順）
- **`.` と `..`**
  - `.` = カレント（今いるところ）
  - `..` = 一つ上

**ハンズオン③：作業フォルダで CRUD**

(1) ターミナルで作業フォルダを作る
```bash
cd ~                  # ホームに移動
mkdir cli-handson     # フォルダを作る
cd cli-handson        # 移動
pwd                   # 今ここを確認
```

(2) テキストエディタで `memo.txt` を作って保存
- 中身：`Hello CLI`
- 保存先：`~/cli-handson/memo.txt`

(3) ターミナルに戻って操作
```bash
ls                    # memo.txt が見えるはず
cat memo.txt          # 中身を表示
cp memo.txt note.txt  # コピー
mv note.txt log.txt   # 名前変更（または移動）
rm log.txt            # 削除
ls
```

> **注**：リダイレクト `>` でファイルを作る方法もあるが、今日はスコープ外。
> エディタ操作とCLI操作を分けて行うフローに統一する。

**重要：`rm -rf` の話（必ずやる、1分）**
- `rm` は**ゴミ箱を経由しない**。即削除
- `rm -rf /` は絶対やってはいけない（PCが死ぬ）
- 「rmする前に `ls` で確認する癖」を強調

**おまけ：`tree` でフォルダ構造を一発で見る（30秒）**
```bash
tree ~/cli-handson
```
- Mac は `brew install tree` で入れる、Git Bash は標準搭載

**GUI⇄CLI の橋渡し：`open .` / `explorer .`（1分）★**
- Mac: `open .` で今のフォルダを Finder で開く
- Windows (Git Bash): `explorer .` で今のフォルダをエクスプローラーで開く
- 「CLIで作業 → GUIで確認」の往復が一瞬になる
- 「CLI と GUI は対立じゃない、使い分けて行き来するのが最強」と伝える

**避けるべき脱線**
- `find` `grep` → 次のセクション4で扱う（ここでは出さない）
- シンボリックリンク → 今日は触れない

**FAQ想定**
- Q「Windowsのフォルダ区切りは `\` じゃない？」→ A「Git Bash では Mac と同じ `/` を使います。今日は `/` で統一」
- Q「日本語ファイル名・スペース入りファイル名は？」→ A「動きます。スペースは `"my file.txt"` のようにクオートで囲む」

### 4. 探す：`find` と `grep`（5分）★ハンズオン④ — AIエージェントの心臓部

**話すポイント**
- オープニングで触れた「**Claude Codeが裏側でやっていること**」=「ファイルを探して、中身を読んで、必要なものを見つける」
- これを人間がやるためのコマンドが `find`（ファイル名で探す）と `grep`（中身で探す）
- **この2つができると、CLIの世界が一気に広がる**

**(1) `find` — 名前で探す**
```bash
find . -name "memo.txt"         # カレント以下から memo.txt を探す
find ~ -name "report.txt"       # ホーム以下から report.txt を探す
```

- `.` = カレントから / `~` = ホームから
- `-name` の後ろに**探したいファイル名**
- 出てくるのは **見つかった場所のパスの一覧**
- `ls` との違い：`ls` は「今ここだけ」、`find` は「サブフォルダも全部」

> **注**：ワイルドカード（`*`）は今日はスコープ外。応用編で扱う。
> 今日は「ファイル名をそのまま指定して、サブフォルダ含めて探す」を体感する。

**(2) `grep` — 中身で探す**
```bash
grep "Hello" memo.txt           # memo.txt の中で "Hello" を含む行を表示
grep -r "TODO" .                # カレント以下のファイル全てを再帰検索
grep -i "error" log.txt         # 大文字小文字を無視
grep -n "import" hello.py       # 行番号付きで表示
```

- `-r` （recursive）= フォルダ以下を全部見る
- `-i` （ignore case）= 大文字小文字を区別しない
- `-n` （number）= 行番号を付ける
- オプションは複数組み合わせOK：`grep -rin "todo" .`

**Windows補足**
- Git Bash 前提なので、`find` も `grep` も Mac とまったく同じく動く
- 講座中の追加説明は不要

**ハンズオン④：ファイルを探す・中身を探す**

(1) エディタで2つファイルを追加（セクション3で作った `memo.txt` に加えて）
- `~/cli-handson/task.txt`　中身：`TODO: 後で書く`
- `~/cli-handson/greeting.txt`　中身：`Hello again`

(2) ターミナルで探してみる
```bash
cd ~/cli-handson

# 名前で探す（フルファイル名でOK。findはサブフォルダも見てくれる）
find . -name "memo.txt"
find . -name "task.txt"

# 中身で探す（具体的なファイル名を指定）
grep "Hello" memo.txt
grep "Hello" greeting.txt
grep -r "TODO" .
```

> **注**：リダイレクト `>` とワイルドカード `*` は今日はスコープ外。
> ファイル作成はエディタで、find/grep は具体的なファイル名を指定する形で体感する。

**ここで一言**：
> 「今やったこれ、Claude Code が裏で動くたびに何百回もやっています。あなたが今、AIの心臓部と同じことをしました」

**避けるべき脱線**
- ワイルドカード `*` `?` → 今日はスコープ外。応用編で扱う
- 正規表現の話 → 一切触れない
- `find -exec` → 応用編
- `sed` `awk` → 名前だけ紹介、ハンズオンなし
- リダイレクト `>` `>>` → 今日はスコープ外。応用編で扱う

**FAQ想定**
- Q「`find` と `ls` の違いは？」→ A「`ls` は今ここの一覧。`find` はサブフォルダも含めて再帰的に検索」
- Q「（参加者がうっかり PowerShell を開いてしまった場合）`find` `grep` が思った動きをしないんですが」→ A「PowerShell を閉じて、Git Bash を開き直してください。今日は Git Bash 前提です」

### 5. 環境変数（4分）★ハンズオン⑤

**話すポイント**
- 環境変数 = PC全体に共有される「設定メモ」
- 代表例：`PATH` = "コマンドを探しに行く場所のリスト"

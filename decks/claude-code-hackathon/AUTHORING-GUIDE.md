# Marp 変換オーサリングガイド

このプロジェクトは Astro ドキュメントサイトの各ページを **Marp スライド** に変換したものです。
スライドは 16:9（1280×720px）固定サイズ。**1枚に収まらない量を入れるとレイアウトが崩れる**ため、
以下のルールを厳守して「崩れない」スライドを作る。

## 1. フロントマター（各ファイル先頭・必須）

```yaml
---
marp: true
theme: hackathon
paginate: true
size: 16:9
footer: '<フッターラベル>'
---
```

`<フッターラベル>` はファイルごとに指定されたものを使う（例 `02 · インストール & セットアップ`）。

## 2. スライド区切り

- スライドの区切りは、行頭の `---`（前後に空行）。
- 各スライドは **1つの話題** に絞る。元の `##` 見出し1つ ≒ スライド1枚〜複数枚。

## 3. テーマのレイアウトクラス

スライド冒頭に HTML コメントでクラスを指定する。

| 指定 | 用途 |
|---|---|
| `<!-- _class: lead -->` | **表紙**（各デッキ1枚目）。濃紺グラデ背景・白文字・中央寄せ |
| `<!-- _class: section -->` | 章の区切り扉（任意）。濃紺背景・白文字 |
| `<!-- _class: compact -->` | やや密度の高いスライド（文字小さめ・余白圧縮） |
| `<!-- _class: dense -->` | 行数の多い表専用（さらに文字小） |
| （無指定） | 通常スライド |

表紙には合わせて次も指定する：

```markdown
<!-- _class: lead -->
<!-- _paginate: false -->
<!-- _footer: '' -->
```

## 4. 1スライドあたりの分量上限（崩れ防止の絶対ルール）

通常スライド（無指定, 本文23px）の目安：

- **本文の行数：合計 12 行以内**（箇条書き・段落あわせて）
- **表：ヘッダー + データ 7 行まで**（3〜4列）
- **コードブロック：12 行まで**（タイトル＋短い説明文と同居する場合は 8 行まで）
- タイトル(`##`)＋短い導入文＋表(7行) のような組み合わせは1枚でOK
- 迷ったら **分割する**。空白が余るのは正義、はみ出しは厳禁。

`compact` クラスなら：本文16行 / 表10行 / コード16行まで。
`dense` クラスなら：表14行まで（FAQ一覧・トラブル一覧などの大きな表向け）。

**表が8行以上のときは `compact` か `dense`、それでも入らなければ表を2枚に分割する。**

## 5. 見出しの使い方

- `#`（h1）：表紙のデッキタイトルのみ。
- `##`（h2）：各コンテンツスライドの**スライドタイトル**（アクセント色＋下線が付く）。1スライドに1つ。
- `###`（h3）/`####`（h4）：スライド内の小見出し。

元ドキュメントで `##` だった見出しが長い手順を含む場合、
`## 大見出し（その1）` / `## 大見出し（その2）` のように分割して複数スライドにする。
（例：「手順（3ステップ）」が長ければステップごとにスライドを分ける）

## 6. 注釈・Tips（引用 `>`）

元ドキュメントの `> ...` は、テーマで**ライトブルーの注釈ボックス**になる。そのまま `>` で書く。
1スライドに注釈ボックスは1個まで（長い注釈は本文化するか別スライドへ）。

## 7. リンクの扱い

- **外部URL**（`https://...`）はそのまま Markdown リンクで残す。
- **サイト内リンク**（`/02-installation` や `/participant-qa` や `/facilitator-troubleshooting` など）は
  スライドではリンクとして機能しないので、**プレーンな参照テキスト**に置き換える。
  例：`[インストール手順](/02-installation)` → `→「インストール & セットアップ」を参照`
  例：`[トラブルシューティング](/facilitator-troubleshooting)` → `→「トラブルシューティング」を参照`

## 8. コードブロック

- 言語指定（```powershell, ```bash 等）はそのまま残す。
- 長い1行コマンド（PATH設定など）は折り返し表示されるので**改変しない**。
- ASCII アート（許可ダイアログの罫線枠など）は ``` でそのまま囲む。崩れる場合は `compact` を使う。

## 9. 内容の保持

- **情報を落とさない**。元ページの全項目をスライドに反映する（表のセル・手順・注意書きすべて）。
- 文面はスライド向けに簡潔化してよいが、意味・数値・コマンドは変えない。
- 体言止め・箇条書き化など、スライドとして読みやすい調整は歓迎。

## 10. 各デッキの構成テンプレート

1. **表紙**（`lead`）：`#` デッキタイトル ＋ `##` サブタイトル（セクション名）＋ 一言説明。
2. （任意）**アジェンダ/概要**スライド。
3. 本編スライド群（`##` ごと、必要に応じ分割）。
4. （任意）**まとめ**スライド。

## 11. 検証（必須）

書いたら必ずレンダリングして**画像で崩れを確認**する：

```bash
cd /Users/kunihiko.haga/Develop/kata-k3s/_SANDBOX/ClaudeCodeHackathon-marp
export CHROME_PATH="/Applications/Google Chrome.app/Contents/MacOS/Google Chrome"
npx -y @marp-team/marp-cli@latest --config-file .marprc.yml --images png slides/<ファイル名>.md -o build/preview/<名前>.png
```

生成された各 PNG を **すべて目視**し、次があれば該当スライドを分割して再レンダリング：
- スライド下端で文字・表・コードが**切れている**（はみ出し）
- 表やコードが**右端をはみ出している**
- 要素が詰まりすぎて読めない

崩れがゼロになるまで繰り返す。

# Claude Code とは？

## セクション 1 ／ Claude Code ハッカソン

PCで実際にアクションを起こせる Claude を 5 分で理解する

---

## ブラウザ版 Claude との違い

| 観点 | ブラウザ版 Claude（claude.ai） | Claude Code |
|---|---|---|
| 役割 | **相談相手**（会話のみ） | **実行できるアシスタント** |
| ファイル操作 | できない | 作成・編集・整理が可能 |
| データ処理 | 限定的 | Excel / CSV / PDF を直接操作 |
| 自動化 | できない | スクリプト作成から実行まで全自動 |

---

## ポイント

- **ターミナルに日本語で指示するだけ。** コマンドを覚える必要はありません
- **ファイルサイズ制限なし・コンテキスト切れなし**、長時間タスクもOK
- **何をするか事前に確認できる** → 安心して使える

> この後のセクションで、インストールから実際の業務効率化までを体験します。
